# 백로그 작성 가이드 — 풀스택 아키텍트 트랙

> Notion 백로그를 만들 때 항상 이 규칙을 따른다.
> 목적: 레이어별로 깊이 있는 아키텍처 결정 경험을 쌓는다.

---

## 핵심 원칙: **1 백로그 = 1 레이어**

한 백로그는 반드시 **Frontend / Backend / Infra 중 하나**의 레이어로만 작업한다.

### 왜 분리하는가
- 한 백로그에 여러 레이어를 섞으면 "이 작업의 아키텍처 결정 포인트"가 흐려진다.
- 풀스택 아키텍트가 되려면 각 레이어의 **고유한 결정 패턴**을 따로 익혀야 한다 (혼합으로는 깊이가 안 생김).
- 코치 prompt 자동 매칭이 작동하려면 레이어가 명확해야 한다.

### 레이어 경계가 모호할 때
"프론트와 백엔드를 잇는 작업"은 별도 카테고리로 만들어 분리한다:
- API 계약 설계 → `[Backend]` 백로그로 (서버가 정의하는 인터페이스)
- 클라이언트 통신 레이어 → `[Frontend]` 백로그로 (클라이언트가 소비하는 방식)

즉, **하나의 작업을 두 백로그로 쪼갠다**.

---

## 백로그 작성 템플릿

Notion 백로그 페이지 본문에 다음 형태로 작성한다.

```markdown
## [Frontend] / [Backend] / [Infra] 작업 제목

**목표**: 한 줄로 무엇을 만드는지

### 아키텍처 결정 포인트 (학습 포인트)
- [ ] 결정 1: 예) 폴더 구조를 feature 단위 vs layer 단위로?
- [ ] 결정 2: 예) 상태 관리 라이브러리 선택 (zustand / jotai / context)
- [ ] 결정 3: ...

### 인터페이스 (다른 레이어와의 접점)
- 의존하는 위/아래 레이어: 예) Backend의 /api/payments 응답 스키마
- 변경 시 영향 받는 곳: 예) 인증 미들웨어, 에러 핸들러

### 검증 (Done 기준)
- [ ] 코드 동작 확인
- [ ] 결정 포인트마다 "왜 이 선택을 했는지" 한 줄 메모 남김
- [ ] 다음에 같은 패턴 적용할 곳 1개 이상 식별
```

---

## Notion 속성 작성 규칙

| 속성 | 작성 규칙 |
|---|---|
| **Title** | `[레이어] 동사 + 대상` 형태. 예: `[Frontend] 결제 페이지 라우팅 설계` |
| **ID** | `BL-001`부터 순차. CLI에서 `coach BL-042` 형태로 호출 |
| **Layer** | **단일 선택**. Frontend / Backend / Infra 중 1개만 |
| **Tags** | 2~4개. Coach Prompts DB와 동일한 옵션 풀 사용 |
| **Status** | Todo / Doing / Done |

⚠️ Layer를 multi-select로 두지 말 것. 멀티 선택을 허용하면 "1 백로그 1 레이어" 원칙이 첫날부터 무너진다.

## 좋은 백로그 vs 나쁜 백로그

### ✅ 좋은 예
- `[Backend] 결제 API 에러 응답 표준화`
  - 레이어 명확, 결정 포인트 좁음 (응답 스키마 형태, HTTP 상태 코드 매핑 전략)
- `[Frontend] 인증 상태 전역 관리 구조 설계`
  - 한 레이어, 학습 포인트 분명 (상태 관리 라이브러리 선택, 토큰 갱신 흐름)
- `[Infra] 환경 변수 관리 전략 도입`
  - 인프라 레벨, 결정 포인트 한정적 (.env 분리 vs secret manager)

### ❌ 나쁜 예
- `결제 기능 만들기`
  - 레이어 표시 없음, 너무 큼, 결정 포인트 모호
- `로그인 구현 (FE+BE)`
  - 혼합 — 두 백로그로 쪼개라
- `버그 수정`
  - 학습 포인트 없음. 단순 버그면 백로그 만들지 말고 그냥 고쳐라

---

## 백로그를 쪼개야 할 신호

다음 중 하나라도 해당하면 **여러 백로그로 쪼개라**:

- 제목에 "and" / "&" / "+" 가 들어감
- Layer를 2개 이상 골라야 할 것 같음
- 아키텍처 결정 포인트가 5개 이상
- 끝나는 데 일주일 이상 걸릴 것 같음

---

## 일일 작업 흐름

```
1. 오늘 작업할 백로그 1개 선택 → Status: Doing
2. 터미널에서 coach BL-XXX 실행
   → 해당 백로그의 Layer/Tags에 매칭되는 Coach Prompts 자동 로드
   → Base + Topic 합쳐 system prompt로 들어감
3. Claude와 함께 아키텍처 결정 → 구현
4. 결정 포인트마다 "왜?" 한 줄씩 백로그 페이지에 메모
5. 작업 끝 → Status: Done
```
